## Auth
- Passwords use per-user-salt `scrypt` hashing (`hashPassword`/`verifyPassword`), with a backward-compatible legacy SHA-256+static-salt verification path for pre-existing hashes. New hashes are always scrypt.
- Session token cookie/header key is `cw_token`; tokens are stored in a `sessions` table (not JWT).
- Register/change-password endpoints require a `confirmPassword` field matching `newPassword`/`password` — this is enforced server-side (`ChangePasswordInput`/register schema), not just client-side.
- Login has a rate limiter (10 attempts/15min per ip+phone) in `rateLimiter.ts`.

**Why:** static-salt SHA-256 and unlimited login attempts were security gaps found during a full security review; fixed without breaking existing sessions.

## API contract / codegen
- Frontend types are generated by Orval from `lib/api-spec/openapi.yaml` into `lib/api-zod/src/generated/types/*.ts` — always treat these as source of truth and fix frontend call sites to match, never hand-edit generated files.
- The admin support-messages endpoint is not covered by codegen; it's called manually in `admin/support.tsx` — its actual shape is `{ userId, data: { message } }` (userId is a path/mutation param, not inside `data`).
- `AdminStats` real field names: `onlineUsers`, `todayDeposit`, `todayWithdrawal`, `totalRevenue`, `totalProfit`, `pendingDeposits`, `pendingWithdrawals` — there is no `activeUsers`/`totalDeposits`/`activeInvestments`/`totalProfitPaid`. Frontend had drifted from this contract; verify against generated types before trusting existing UI code.

## Environment / workflows
- This project has no traditional `.replit` `[[workflows.workflow]]` entries — workflows are auto-derived from each artifact's `.replit-artifact/artifact.toml` `[[services]]` blocks. Use `listWorkflows()` (code_execution) to get the real runtime workflow names (format: `artifacts/<slug>: <service name>`) rather than guessing names like "API Server".
- After a fresh/rebuilt DB, Drizzle schema is not auto-applied — run `pnpm --filter @workspace/db run push` (drizzle-kit push) before the API server can serve any request that touches the DB, or every route 500s with "relation does not exist".

**Why:** first boot of the api-server workflow threw `relation "users" does not exist` on register — schema push had never been run despite `DATABASE_URL` already being set.
